🧬 TMA Formator

Référentiel opérationnel pour la gestion quotidienne de la TMA Ivalua.

📖
Chapitre 0
Introduction & périmètre
À qui s'adresse ce document et comment l'utiliser
🎯 Objectif
Constituer la référence unique pour tout consultant TMA Ivalua. Ce document couvre les processus, conventions et bonnes pratiques à respecter sur chaque mission.
👥 Audience
Consultants, chefs de projet TMA, partenaires. Toute personne intervenant sur le périmètre Ivalua Cyrias.
🔄 Mise à jour
Ce référentiel est vivant. Toute évolution de processus doit être répercutée ici avant déploiement.
⚠️ Règle d'or : Toutes les demandes IT soumises à l'équipe Ivalua doivent être rédigées en anglais, sans exception.
📋
Chapitre 1
Gestion des demandes & tickets
Cycle de vie d'une demande, de la réception à la clôture
📥 Réception
Chaque demande est enregistrée dans l'outil de ticketing (ServiceNow / Jira). Vérifier : client, priorité, description fonctionnelle complète, environnement cible (DEV / UAT / PROD).
🔍 Qualification
Analyser la demande sous 24h ouvrées. Identifier : type (bug / évolution / paramétrage), criticité, impact périmètre, estimation charge. Revenir vers le client si besoin de clarification.
⚙️ Réalisation
Développement ou paramétrage en DEV. Tests unitaires obligatoires. Validation en UAT avec le client avant toute mise en production.
✅ Clôture
Mise en production planifiée lors d'une fenêtre de déploiement validée. Mise à jour du ticket (statut, commentaire de résolution, version déployée). Notification client.
⏱️ SLA standard : Critique → 4h / Haute → 1j / Moyenne → 3j / Basse → 5j. Ces délais courent à partir de la qualification du ticket.
🛠️
Chapitre 2
Modifications SQL & base de données
Règles et checklist pour toute intervention en base Ivalua
📝 Rédaction du script
Utiliser le Générateur SQL intégré (onglet ⚡). Inclure systématiquement : entête avec N° demande, date, auteur — BEGIN TRANSACTION / ROLLBACK / COMMIT — SELECT de contrôle avant et après modification.
🔒 Règles obligatoires
Toujours utiliser des alias de table. Jamais de UPDATE/DELETE sans clause WHERE. Utiliser NOLOCK sur les SELECT de contrôle. Limiter les batchs à 500 lignes max pour les gros volumes.
📋 Checklist validation IT
34 critères à valider avant soumission (cf. Générateur SQL). Points clés : SET NOCOUNT ON, gestion des erreurs TRY/CATCH, pas de curseurs, pas de SELECT *, index hints si nécessaire.
🚀 Déploiement
Soumettre via le portail IT Ivalua (en anglais). Joindre le script validé + le résultat du SELECT de contrôle. Déploiement assuré par l'équipe IT Ivalua uniquement (pas d'accès direct PROD).
🚫 Interdit : Aucun accès direct en PROD sans validation du client ET de l'IT Ivalua. Toute modification non tracée sera annulée.
📊
Chapitre 3
Saisie des CRA & suivi des temps
Règles de saisie Stafiz et bonnes pratiques
🗓️ Fréquence
Saisie quotidienne recommandée, hebdomadaire obligatoire. Clôture du CRA le vendredi soir ou au plus tard le lundi matin suivant. Aucun rattrapage au-delà de 2 semaines.
💬 Commentaires
Chaque ligne CRA doit avoir un commentaire explicite : action réalisée, ticket associé (N° demande), livrable. Éviter les commentaires vides ou génériques ("Développement", "Support").
🏷️ Granularité
Utiliser les tâches Stafiz configurées par mission : Développement / Paramétrage / Tests / Recette / Coordination / Documentation / Support. Minimum 0,25 jour par ligne.
📈 Suivi budget
Alerter le chef de projet si le RAC (Reste À Consommer) tombe sous 20% du budget alloué. Le CRAminator génère ces alertes automatiquement.
🔄
Chapitre 4
Environnements & déploiements
Gestion des environnements Ivalua et processus de mise en production
🏗️ DEV
Environnement de développement et tests unitaires. Accès direct Cyrias. Données anonymisées. Rafraîchissement depuis PROD tous les 3 mois (à confirmer par client).
🧪 UAT / Recette
Environnement de recette client. Tests d'acceptance obligatoires avant toute MEP. Validation formelle (email ou ticket) exigée du référent client. Gel des déploiements 48h avant MEP PROD.
🚀 PROD
Déploiements uniquement sur fenêtres planifiées (généralement hors heures de bureau). Rollback préparé systématiquement. Communication préalable aux utilisateurs si impact fonctionnel.
📋 Change Request
Toute MEP PROD nécessite un CR validé. Soumettre via le portail IT Ivalua en anglais avec : description du changement, impact, plan de rollback, résultats de recette UAT.
🗓️ Fenêtres de déploiement : À confirmer avec chaque client. En général : mardis et jeudis, 18h–20h (hors période de clôture comptable).
🤝
Chapitre 5
Communication client
Bonnes pratiques relationnelles et points de suivi
📅 Points hebdomadaires
Organiser un point hebdomadaire avec chaque client actif (30–60 min). Ordre du jour type : tickets en cours / avancement / blocages / planning semaine suivante. Compte-rendu envoyé dans les 24h.
📧 Emails & communications
Répondre à tout email client sous 4h ouvrées. Pour les demandes complexes : accuser réception immédiatement, donner une estimation de délai de traitement. Copier le chef de projet sur les échanges sensibles.
⚠️ Gestion des escalades
En cas de blocage ou mécontentement client : escalader immédiatement au chef de projet. Ne jamais faire de promesses sans validation interne. Documenter les incidents dans le suivi hebdo (CRAminator).
📊 Reporting mensuel
Produire un rapport mensuel par client : consommation vs budget, tickets traités/ouverts, actions du mois, plan du mois suivant. Utiliser l'export PPTX du CRAminator comme base.
⚡
Chapitre 6
Paramétrage Ivalua
Conventions et bonnes pratiques de configuration
🏷️ Conventions de nommage
Préfixer les objets créés par Cyrias : CYRIAS_ ou CYR_. Utiliser l'anglais pour tous les identifiants techniques. Documenter chaque paramétrage dans la fiche technique du ticket associé.
🔧 Workflows & approbations
Tout nouveau workflow doit être validé en DEV + UAT avant PROD. Tester toutes les branches (approbation, refus, délégation, escalade automatique). Documenter les rôles et les seuils.
📑 Formulaires & champs
Minimiser les champs obligatoires (impact adoption utilisateur). Nommer les champs en cohérence avec le vocabulaire client. Prévoir la traduction si environnement multilingue.
🔗 Interfaces & connecteurs
Documenter systématiquement : format des fichiers, fréquence, gestion des erreurs, logs de monitoring. Tester les cas nominaux ET les cas d'erreur en UAT.
📝 Documentation : Tout paramétrage sans documentation associée sera considéré comme incomplet. La doc technique est livrée en même temps que le développement.

Aide — CRAMINATOR

1 Importer vos données

Au lancement, déposez un ou plusieurs fichiers .xlsx ou .csv exportés depuis Stafiz dans la zone de dépôt. Vous pouvez aussi coller un lien Google Sheets public.

Colonnes attendues : Date, Collaborateur, Equipe, Client, Mission, Tâche, Jours (ou Fractions), Commentaire.
Un bouton « Télécharger un modèle » est disponible sur l'écran d'import.

2 Dashboard

L'onglet principal affiche une vue synthétique de votre activité TMA :

Graphiques — Jours par client, par collaborateur, répartition par tâche, évolution temporelle, CA estimé, qualité de saisie.
Tableaux — Suivi Budgets & RAC (Reste À Consommer) et Taux de production (TACE) par collaborateur.
Alertes — Missions dont le RAC ≤ 10% et collaborateurs en surcharge ou sous-charge.
IA — L'assistant analyse vos données et génère une synthèse avec recommandations.

3 Filtres avancés

Cliquez sur Filtrer pour ouvrir le panneau de filtres en cascade. Les filtres sont hiérarchiques : la sélection d'une Année restreint les Semestres disponibles, etc. Les filtres opérationnels (Équipe → Client → Mission → Collaborateur) suivent la même logique.

4 Fiches Clients

Chaque client dispose d'une fiche d'identité complète :

• Informations commerciales — Contact, email, contrat, responsable interne (modifiables, sauvegardées en local).

• Indicateurs — TJM, jours consommés, CA estimé, équipes, nombre de collaborateurs.

• Graphiques — Répartition par tâche et évolution des jours.

• Export PPTX — Générez une fiche PowerPoint du client en 1 clic.

5 Explorateur de données

Un tableau détaillé de toutes les lignes CRA avec :

• Filtres locaux — Client, Mission et Mois (en plus des filtres globaux du Dashboard).

• Recherche texte — Recherchez dans les commentaires, clients, collaborateurs, tâches…

• Export Excel — Exporte uniquement les données visibles (filtrées) au format .xlsx.

6 Suivi Hebdo

Un carnet de notes pour le suivi hebdomadaire par client :

• Saisie — Sélectionnez un client, une semaine, une catégorie (Point général, Alerte, Décision, Action, Satisfaction, Staffing…) et rédigez votre commentaire.

• Historique — Tous les commentaires sont affichés chronologiquement avec badges colorés par catégorie.

• Export Excel — Exportez l'historique complet ou filtré par client.

• Les notes sont sauvegardées localement dans votre navigateur.

7 Demandes / Tickets

Un module complet pour suivre les demandes et tickets clients, avec import dédié :

• Import Excel — Importez un fichier .xlsx contenant vos tickets. Colonnes attendues : ID, Date, Client, Objet, Statut, Priorité, Satisfaction (Satisfait / Neutre / Insatisfait). Un bouton « Modèle » génère un template.

• Statuts personnalisables — Ajoutez, renommez, colorez et supprimez des statuts. Cochez « Fermé » pour les statuts clôturés (pris en compte dans le taux de clôture).

• KPI automatiques — Total, Ouvertes, Résolus, Taux de clôture, % Satisfaits et % Insatisfaits.

• 3 graphiques — Répartition par statut, volume par client, distribution de la satisfaction.

• Tableau filtrable — Filtrez par Client, Statut et Satisfaction. Export Excel des données filtrées.

• Les données sont sauvegardées en local (indépendantes des CRA).

8 TJM & Budgets

Depuis la barre d'outils :

• TJM — Définissez le Taux Journalier Moyen par client. Le CA estimé se calcule automatiquement partout.

• Budgets — Attribuez un budget (en jours) à chaque mission et cochez « Fermé » pour archiver les missions terminées (exclues des calculs).

9 Exports

• Excel — Exporte les données filtrées du Dashboard, de l'Explorateur ou des Demandes.

• PPTX — Génère un PowerPoint personnalisable : choisissez les graphiques et les fiches clients à inclure. Design professionnel aux couleurs Cyrias.

• Charts individuels — Survolez un graphique pour voir le bouton de téléchargement PNG.

10 Affichage & Données

• Affichage — Masquez/affichez les blocs du Dashboard (graphiques, alertes, tableaux, IA).

• Multi-sources — Ajoutez plusieurs fichiers CRA. Chaque source apparaît comme un chip coloré supprimable.

• Persistance — Toutes les données (sources CRA, tickets, TJM, budgets, profils clients, notes de suivi, statuts personnalisés, préférences d'affichage) sont sauvegardées dans le navigateur (localStorage).

Astuce : Toutes les données restent 100% en local dans votre navigateur. Aucun envoi vers un serveur (sauf l'analyse IA qui utilise l'API Gemini).
